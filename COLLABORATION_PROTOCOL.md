# 人机协作协议：ReAct 式任务分解与迭代执行

> **本文档的作用：** 任何角色（人类、规划 Agent、执行 Agent）在加入协作前阅读此文档，即可理解整体背景、协作模式、通信方式和执行规范。无需额外口头补充。
> **版本：** 2026-04-10 · 基于 ragent 项目 Agent Loop 合入实践总结

---

## 一、核心理念

我们采用一种**抽象的 ReAct（Reason + Act）循环**来完成复杂的软件工程任务。核心思想是：

- **思考与执行分离**：规划者负责"想清楚做什么"，执行者负责"把事做对"
- **文件即通信**：所有任务指令和执行反馈通过文件传递，不依赖口头沟通或复制粘贴
- **信息搜集优先于动手**：在修改代码之前，先让执行者摸清现状，基于事实而非假设做决策
- **最小可行实现**：每一步追求"跑通"而非"完美"，复杂问题分多轮迭代解决
- **最大化复用**：优先利用系统中已有的组件和模式，不做不必要的重新发明

---

## 二、角色定义

### 操作者（Operator）— 人类

- 发起需求，提供项目背景和领域知识
- 在规划者和执行者之间传递文件（TASK → CC，FEEDBACK → 规划者）
- 补充执行者无法获取的信息（如线上环境状态、业务判断）
- 在规划者可能陷入局部最优时提供全局视角和新方向
- 拥有最终决策权

### 规划者（Planner）— Claude（对话 Agent）

- 与操作者讨论需求，进行任务分解
- 将讨论结论转化为结构化的 TASK 文件
- 阅读执行者的 FEEDBACK/REPORT，分析结果，决定下一步
- 维护整体进度和方向感，确保不偏离主线
- 在必要时调整策略（基于执行反馈，而非坚持初始计划）

### 执行者（Executor）— Claude Code（CC）

- 阅读 TASK 文件，理解目标和约束
- 在实际代码环境中执行任务（调查、编码、测试、部署）
- 将执行结果整理为结构化的 REPORT 文件
- 拥有执行层面的灵活性：实际情况比理想复杂，CC 可根据现场判断选择最佳实现路径
- **可以反问**：当遇到 TASK 描述与实际不符、信息不足以做决策、或发现更好的替代方案时，CC 应在 REPORT 中明确提出疑问和建议，而非盲目执行

## 三、ReAct 循环流程

```
┌──────────────────────────────────────────────────────┐
│                                                      │
│   操作者 + 规划者                                     │
│   ┌─────────────┐                                    │
│   │  讨论 & 规划  │                                    │
│   └──────┬──────┘                                    │
│          │ 产出 TASK 文件                              │
│          ▼                                           │
│   ┌─────────────┐     文件传递      ┌──────────────┐ │
│   │  TASK 目录   │ ──────────────→ │  执行者 (CC)  │ │
│   └─────────────┘                  └──────┬───────┘ │
│                                           │          │
│                                    执行 & 记录        │
│                                           │          │
│   ┌──────────────┐     文件传递     ┌─────┴────────┐ │
│   │ 操作者 + 规划者│ ←──────────── │ FEEDBACK 目录 │ │
│   └──────┬───────┘               └──────────────┘  │
│          │                                           │
│          │ 分析结果，决定下一步                         │
│          │                                           │
│          └───────── 继续循环或结束 ──────────────────→ │
│                                                      │
└──────────────────────────────────────────────────────┘
```

**每一轮循环的典型节奏：**

1. **Reason**：操作者提出需求或反馈 → 规划者分析、讨论、形成 TASK
2. **Act**：操作者将 TASK 传递给 CC → CC 执行
3. **Observe**：CC 产出 REPORT → 操作者传回给规划者 → 规划者分析结果
4. **Decide**：基于结果决定——继续下一个 TASK？调整方向？还是任务线完成？

## 四、文件通信规范

### 目录结构

```
项目根目录/
├── TASK/                          # 待执行的任务文件
│   ├── CC_Task_001_Investigation.md
│   ├── CC_Task_002_Implementation.md
│   └── ...
├── FEEDBACK/                      # 执行者的反馈/报告
│   ├── CC_Report_001_Investigation.md
│   ├── CC_Report_002_Implementation.md
│   └── ...
└── COLLABORATION_PROTOCOL.md      # 本文档
```

### TASK 文件规范

文件命名：`CC_Task_<序号>_<简要描述>.md`

结构模板：

```markdown
# CC Task <序号>：<任务标题>

> **目标：** 一句话说清楚这个任务要达成什么
> **约束：** 明确的边界（只读不改 / 最小化改动 / 不动某个模块）
> **产出：** CC 需要生成的 REPORT 文件名

---

## 背景（为什么要做这个）

简要说明这个任务在整体计划中的位置，以及它依赖的前置信息。

## 具体步骤

### 1. 第一步
具体要做什么，检查什么，产出什么。

### 2. 第二步
...

## 产出格式要求

REPORT 文件的结构模板（让 CC 知道我们需要什么信息）。

## 注意事项

- 如果发现实际情况与描述不符，以实际代码为准，在 REPORT 中说明差异
- 如果信息不足以做某个决策，在 REPORT 中提出问题，不要猜测
- 如果发现更好的替代方案，在 REPORT 中建议，但不要擅自偏离任务目标
```

### REPORT 文件规范

文件命名：`CC_Report_<序号>_<与 TASK 对应的描述>.md`

关键原则：
- **事实与判断分离**：先列客观发现（文件路径、方法签名、实际行为），再给出主观评估
- **问题要具体**：如果遇到阻碍或需要决策输入，描述清楚"我看到了什么、我不确定什么、我需要知道什么"
- **改动要可追溯**：如果修改了代码，列出每个文件的变更和原因

## 五、执行原则

### 对执行者（CC）的指导

1. **先理解再动手**：拿到 TASK 后，先通读理解目标和约束，再开始执行。如果有不清楚的地方，在 REPORT 开头提出。

2. **最小化实现**：追求能跑通的最简方案。不做过度设计、不做提前优化、不做 TASK 没要求的事。

3. **最大化复用**：优先使用项目中已有的组件、服务、模式。新建类/方法只在必要时才做。

4. **实际代码为准**：TASK 中的伪代码和假设可能与实际不符（因为规划者没有看过全部源码）。以实际代码为准，在 REPORT 中说明差异和你的实际做法。

5. **灵活应变**：遇到 TASK 没预料到的问题，CC 有权自行判断最佳解决方式。原则是：不偏离任务目标的前提下，选择最简单可行的方案。在 REPORT 中记录你的判断和理由。

6. **主动反馈**：如果发现以下情况，必须在 REPORT 中明确指出：
   - TASK 的假设与实际不符
   - 有比 TASK 建议的更好的实现方式
   - 需要操作者提供额外信息才能继续
   - 执行中发现的风险或副作用

### 对规划者的指导

1. **信息搜集先行**：对不熟悉的代码库，先出调查类 TASK（只读不改），基于事实再规划实施。

2. **TASK 粒度适中**：一个 TASK 对应一个清晰的目标。太大则 CC 容易迷失方向，太小则来回轮次过多。典型粒度：一次调查、一次实现、一次验证。

3. **给约束不给死板步骤**：告诉 CC "目标是什么、不能做什么"，而非逐行规定怎么写代码。CC 在执行层面比规划者更有信息优势。

4. **快速响应反馈**：拿到 REPORT 后，提取关键信息，快速决策下一步。不要在已完成的细节上纠缠。

5. **保持主线意识**：每轮循环开始前，回顾整体目标，确保当前 TASK 是朝着正确方向前进。

### 对操作者的指导

1. **补充全局视角**：当规划者和执行者都可能陷入局部最优时，提供新的思路、方向或外部信息。

2. **忠实传递**：TASK 和 REPORT 通过文件传递，不做未经讨论的修改。

3. **效率优先**：如果某个小问题自己处理更快（比如改一行配置），直接做，不必走完整的 TASK 流程。

## 六、任务类型模式

经过实践，我们总结了几种常用的任务类型：

### 调查型（Investigation）
- **目的：** 搜集信息，为决策提供依据
- **特征：** 只读不改，产出是事实报告
- **典型场景：** 进入新代码库、评估技术方案可行性、源码调研
- **REPORT 重点：** 文件路径、接口签名、依赖关系、兼容性风险

### 实施型（Implementation）
- **目的：** 编写/修改代码，实现功能
- **特征：** 有明确的设计决策作为输入，产出是可运行的代码
- **典型场景：** 功能开发、bug 修复、重构
- **REPORT 重点：** 文件变更清单、遇到的问题和解决方式、编译/部署结果

### 验证型（Verification）
- **目的：** 确认实施结果是否符合预期
- **特征：** 运行测试、检查日志、验证端到端链路
- **典型场景：** 功能验证、回归测试、性能检查
- **REPORT 重点：** 测试结果、日志证据、发现的问题和修复（如果授权直接修）

### 调研型（Research）
- **目的：** 带着特定目标去大型代码库或技术领域中寻找核心逻辑和设计模式
- **特征：** 有明确的调研问题，但答案可能分散在多个地方
- **典型场景：** 分析开源项目架构、学习新技术方案、竞品调研
- **REPORT 重点：** 核心发现（带代码/文件引用）、架构图/流程图、可借鉴的设计模式、与自身项目的关联

## 七、当前项目背景（ragent）

### 系统概况

ragent 是一个基于 Java/Spring Boot 的 RAG（检索增强生成）系统，部署在腾讯云。包含两个进程：

- **bootstrap**（端口 9090）：RAG 主系统，处理 Pipeline 问答和 Agent Loop 问答
- **mcp-server**（端口 9099）：MCP 协议层，提供工具给外部 Agent 调用

技术栈：Java / Spring Boot / Milvus / 百炼 API / MySQL / Redis

### 已完成的里程碑

| 阶段 | 内容 | 状态 |
|------|------|------|
| Task 001 调查 | 摸清 bootstrap 的 LLMService、Chat 端点、检索服务、AgentLoop 实验代码 | ✅ 完成 |
| Task 002 实施 | 将 AgentLoop 最小化合入 bootstrap，新增 7 个文件 + 1 个修改 | ✅ 完成 |
| Task 003 验证 | 功能验证全部通过，Pipeline 回归正常，Agent 模式可用 | ✅ 完成 |

### 当前能力

- **Pipeline RAG**：传统的检索增强生成，固定流程（查询重写 → 意图路由 → 检索 → Rerank → 生成）
- **Agentic RAG**：Agent Loop 模式，模型自主决定搜索策略和回答时机，通过 `/api/ragent/agent/chat` 端点访问

### 待推进方向

1. **Agent 模式前端集成**：让用户在 web 端能切换使用 Agent 模式
2. **多 Agent 架构调研**：研究 Agent 编排、任务分解、上下文传递等高级能力（参考 claude-code 等开源实现）

## 八、补充说明

### 关于灵活性

本协议是指导而非束缚。具体执行中：

- TASK 的具体步骤是建议，不是死命令——CC 应根据实际情况灵活调整
- 目录结构可以根据项目实际情况调整
- 如果某个任务简单到不需要完整的 TASK/REPORT 流程，可以简化

### 关于迭代

- 不要试图在一个 TASK 中解决所有问题
- 每轮循环的目标是向前推进一步，而非一步到位
- 遇到岔路口时，优先选择可逆的、低风险的方向

### 关于沟通效率

- TASK 写清楚目标和约束，比写详细步骤更重要
- REPORT 提炼关键信息，比堆砌细节更重要
- 如果一句话能说清楚，不要写一段话
