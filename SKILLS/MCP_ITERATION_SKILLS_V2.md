# MCP 系统迭代 V2：评测工具、范式对比与全链路认知

> 承接 MCP_ITERATION_SKILLS_V1（打通 knowledge_search），这一轮的主题是"让 MCP 不只是能用，而是能产出可衡量的价值"。
> 2026-03-31，ragent_qa 工具实现 + 首次定量评测完成。

---

## 一、我们做了什么，为什么这样做

### 起点

MCP V1 打通了 knowledge_search，CC 能查知识库了。但有两个问题悬而未决：第一，裸向量检索（无 rerank、无截断）对 CC 够用吗？第二，面试官会问"你做过数字化评估吗"——我们没有数字。

### 核心决策：新增 ragent_qa 工具

**动机：** 不是替代 knowledge_search，而是让 CC 同时拥有两个不同粒度的信息通道：

| 工具 | 走的路径 | 返回内容 | 适用场景 |
|------|---------|---------|---------|
| knowledge_search | 裸向量检索 | 原始 chunks + scores | 需要原文片段、验证引用 |
| ragent_qa | 完整 RAG 管道（重写→意图→检索→rerank→截断→生成） | 结构化回答 + 引用来源 | 需要完整答案、评测对比 |

**Canal 式的接法：** Ragent 的对话接口 `/rag/v3/chat` 本来就在那里，ragent_qa 只是在 mcp-server 里加了一个 Executor 调这个接口。不改 Ragent 任何代码。唯一的技术点是需要处理 SSE 流式响应——读取所有 `data:` 行拼接成完整文本。

### 踩坑

| 坑 | 原因 | 解决 |
|----|------|------|
| 404 not found | bootstrap 全局路径前缀 `/api/ragent`，和 V1 的 RetrieveController 同一个坑 | 路径改为 `/api/ragent/rag/v3/chat` |

**这个坑踩了两次，说明 SKILLS 的价值——V1 记录过这个前缀问题，V2 立刻定位到根因。**

## 二、首次定量评测

### 评测设计

用同一组 6 个测试问题，分别通过两个 MCP 工具获取结果，CC 同时充当执行者和评判者。

### 评测结果

| 问题 | ragent_qa 正确性 | knowledge_search top1 score | 差异观察 |
|------|-----------------|---------------------------|---------|
| Q1 向量数据库选型 | ✅ 正确 | 0.66 | 完整管道直接给出结论+原因 |
| Q2 查询重写机制 | ✅ 正确 | 0.71 | 完整管道综合多个文档形成系统解释 |
| Q3 手搓 vs SpringAI | ✅ 正确 | 0.68 | 完整管道结构化呈现对比分析 |
| Q4 MCP 协议选型 | ✅ 正确 | 0.81 | 两者都命中，完整管道给出结构化对比 |
| Q5 幻觉抑制 | ✅ 正确 | 0.75 | 完整管道给出方法论，裸检索返回技术细节片段 |
| Q6 前端技术栈 | ✅ 正确 | 0.65 | 简单事实查询差异最小 |

**量化汇总：** ragent_qa 正确回答率 6/6（100%）；knowledge_search 平均 top1 score 0.71，最高 0.81，最低 0.65。

### 关键发现

- **完整管道在"为什么"类问题上优势显著（Q2/Q3/Q5）：** 需要跨文档综合推理的问题，裸检索只能返回分散片段，完整管道能整合成结构化答案
- **简单事实查询差异不大（Q6）：** 说明优化投入应聚焦在复杂问题场景
- **裸检索的 score 分布（0.59~0.81）印证了 V1 的发现：** Q4（MCP 专题）score 最高，Q1（通用选型）score 最低——问题越聚焦，向量检索越准
- **CC 作为评测员是可行的：** 能正确判断回答质量，产出结构化报告，这个流程可重复运行

### 这份数据的三个用途

1. **面试回答 Q4（"做过数字化评估吗"）**：有 6 个测试问题 × 2 个通道的对比数据，有量化指标
2. **基线数据**：后续任何改动（换截断阈值、改分块策略、换 embedding 模型）重跑对比
3. **SFT 数据积累**：每次 ragent_qa 调用产生的 (question, answer, references) 是天然训练样本

## 三、范式对比：Supermemory vs OpenViking vs 我们

### 底层赌注

三个系统回答同一个根本问题——"怎么用最少 token 把最对的信息送进上下文"——但下了不同的赌注：

| 维度 | Supermemory | OpenViking | 我们的系统 |
|------|------------|------------|-----------|
| 核心赌注 | 用 LLM 推理替代向量数学 | 用文件结构替代扁平存储 | 用人工策展 + 向量检索 |
| 写入机制 | AI 响应后 fire-and-forget，LLM 异步提取 | 文件导入时 DAG 异步生成 L0/L1 | 人工写 SKILLS 文档 |
| 路由机制 | containerTag 租户隔离，无语义路由 | 全局向量搜索 seed → 目录优先队列 | 意图树描述 + LLM 分类 |
| MCP 工具 | 2 个（memory / recall） | 16 个（文件系统操作） | 5 个（search/qa + 3 模拟） |

### 关键洞察

**发现一：Supermemory 的后端提取逻辑是黑盒。** 开源仓库里没有 extracting 阶段的 prompt 和 LLM 调用代码。99% 准确率的 ASMR 不在开源部分。

**发现二：OpenViking 的"目录递归检索"和我们的"意图树路由"是同一个问题的两种解法。** OpenViking 用向量分数做路由决策（best-first 树遍历），我们用 LLM 做路由决策（意图分类）。前者更自动化但依赖 embedding 质量，后者更精准但依赖人工配置。

**发现三：我们的 SKILLS 文档本质上是人工版的 L0/L1。** OpenViking 花 LLM token 自动生成目录摘要；我们花人的时间手动写。质量更高，但吞吐量受限于人。

### 未验证的接线点

**接线点 1（最高优先级，未实施）：** 把 knowledge_search 拆成三个按知识库分开的工具，description 复用意图树描述。让 CC 的工具选择能力充当路由器，验证"MCP 工具描述能否替代意图树路由"。

**接线点 2：** knowledge_search 返回值加 score 分布摘要，让 CC 自己做截断决策。

**接线点 3：** 给 SKILLS 文档加 metadata 标签，knowledge_search 支持优先返回 SKILLS 类文档。

## 四、全链路认知（CC 侦察 + MCP 知识库交叉验证）

### 问答链路（后半段）

```
用户提问 → SSE连接 → 记忆加载(MySQL直读) → 查询重写(LLM①) → 意图路由(LLM②, Redis缓存意图树) 
→ 混合检索(Milvus向量+BM25) → Rerank(百炼API) → 自适应截断 → Context格式化+文档标签注入
→ Prompt组装(三种模板) → LLM生成(LLM③~④, 多模型降级+首包探测) → SSE流式返回+引用溯源
```

### 文档入库链路（前半段）

```
上传文档 → S3存储+MySQL记录 → 手动触发分块(分布式锁) → Tika解析 
→ StructureAware分块(targetChars=1400, 不切断标题/代码块) → Embedding(qwen-emb8b) → Milvus入库
```

### 基础设施分工

| 组件 | 角色 | 缓存/存储内容 |
|------|------|-------------|
| MySQL | 持久化层 | 文档元数据、chunk记录、对话历史、意图树配置 |
| Redis | 加速层 | 意图树(7天TTL+手动清除)、docName映射、分布式锁 |
| Milvus | 检索层 | 向量+原文，按docId分组 |
| S3/MinIO | 存储层 | 原始文件 |
| 百炼API | 智能层 | LLM调用(重写/分类/生成) + Rerank |

### 系统的三个"没有"（因果链）

```
没有评测机制 ← 因为 → 没有标注数据集 ← 因为 → 没有自动记录用户案例
```

我们已经在手动补这条链：6个测试问题=手动评测集，分数分布基线=手动指标，SKILLS文档=手动用户案例。ragent_qa 工具是第一步自动化——每次调用自动积累 (question, answer, references) 三元组。

## 七、未完成的线索

**关于评测的深化：**
- 当前评测只有"正确/不正确"的二元判断，缺少细粒度指标（召回率、精确率需要标注数据）
- 可以设计"CC 评判 + 人工抽检"的两阶段评测，先 CC 批量跑，人工只验证边界 case
- ragent_qa 的耗时数据还没收集，这是性能基线的一部分

**关于接线点 1 的实验：**
- 拆分 knowledge_search 为三个工具后，CC 的路由准确率是否等同于意图树？
- 如果等同，意味着对 CC 调用方来说，意图树是多余的中间层——路由逻辑从服务端转移到客户端

**关于自动化飞轮：**
- 理想链路：人策展 SKILLS → 入库 → CC 自己找到 SKILLS → 恢复上下文 → 协作 → 产生新知识 → 人策展新 SKILLS
- 当前断点：CC 不会主动去搜 SKILLS，需要人告诉它"去知识库找之前的记录"
- 可能的解法：CC 启动时的 system prompt 或 CLAUDE.md 里加一句"知识库里有 SKILLS 文档，遇到项目相关问题先搜"

**关于更大的视野：**
- Agent SFT 训练数据的四条路线（强模型蒸馏、线上日志、人工种子+LLM扩写、开源数据集）——ragent_qa 产生的三元组属于"线上日志"路线
- 评测集（100-200条）比训练集更重要——我们的 6 个测试问题是起点，日常使用中的 bad case 应该持续积累进来
