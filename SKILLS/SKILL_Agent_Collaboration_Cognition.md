# 智能体协作认知：从 ragent 实践中提炼的工作原则

> **性质：** 工作假设，不是定论。这些原则来自 ragent 项目中 Agent 模式调试和人机协作的具体实践（Task 009-011），在特定条件下经过验证。条件变化时，结论可能需要修正。
> **更新记录：** 2026-04-12 · 基于 4 个知识库、百炼 API 模型的实践

---

## 一、核心认知：智能体是有思考能力但缺乏上下文的实体

无论是 ragent 中的 Agent、协作中的 CC（Claude Code），还是任何你需要协作的智能体，它们的共同特征是：

- **有推理能力**：能理解目标、分析信息、做出判断
- **缺乏上下文**：不知道你知道的事情，不知道环境的全貌
- **依赖工具**：能力边界由可用的工具和信息决定

因此，与智能体协作的核心问题不是"让它更聪明"，而是**"让它看到做决策所需的信息"**。

这个认知看似简单，但容易犯的错误是：把信息堆上去就以为够了。实际上，信息的**组织方式**比信息的**数量**更重要。

---

## 二、实践原则

### 原则 1：分层信息供给

**表述：** 信息按"索引 → 详情 → 策略"三层组织，密度递减，获取成本递增。

**在 ragent Agent 中的验证：**
- 索引层（工具描述，每库一行）：让 Agent 知道"有什么可选"
- 详情层（system_info_query，按需获取）：让 Agent 深入了解"某个选项的详细信息"
- 策略层（system prompt 路由规则）：让 Agent 知道"怎么做决策、怎么纠错"

**在人机协作中的体现：**
- TASK 的背景段 = 索引（让 CC 知道在做什么）
- 前置阅读的文件 = 详情（让 CC 按需获取完整上下文）
- 协作协议的执行原则 = 策略（让 CC 知道怎么行动）

**边界：** 当前验证规模是 4 个知识库。当规模增长到 50+，索引层的"每个一行"是否仍然有效，需要重新验证。可能需要引入分组、层级索引等机制。

### 原则 2：策略引导优于信息堆砌

**表述：** 教智能体"怎么想"比告诉它"有什么"更有效。

**验证数据：**
- Task 011 中，增强工具描述（信息层面）让 Agent 首次选择准确率有所提升，但仍不完美
- 增强 system prompt 的路由策略（策略层面）带来了更关键的改变——Agent 学会了"低分换库"的自我纠正行为
- 8 行策略规则的效果 > 精心组织的知识库描述

**推论：** 设计 Agent 的 prompt 时，优先投入在行为策略上，其次才是知识注入。好的策略能弥补不完美的信息，反过来不成立。

**边界：** 策略引导的前提是智能体有足够的推理能力来执行策略。如果模型能力不足（如小参数模型），策略引导的效果会打折扣，可能需要更刚性的规则或更丰富的信息补偿。

### 原则 3：单一数据源

**表述：** 同一类信息在系统中只有一个源头，所有消费方从同一个源头读取。

**验证事件：** Task 009 的根因就是 system_info_query（读数据库）和工具验证（硬编码白名单）数据源不一致，导致新增知识库后一边认一边不认。

**推广：** 在 Agent 系统中，任何需要被多个组件引用的信息（知识库列表、工具能力、系统状态），都应该只有一个权威来源。宁可多一次查询，不要维护两份副本。

### 原则 4：先确认现状再行动

**表述：** 不基于假设行动，不基于过时信息行动。

**在 Agent 调试中的体现：**
- Task 010 没有直接猜根因就动手修，而是先调查模型实际看到的信息
- Task 011 要求 CC 在实施前先确认代码现状，不假设 Report 010 的描述仍然准确

**在人机协作中的体现：**
- 调查型 TASK 先于实施型 TASK
- CC 被鼓励在 REPORT 中指出"TASK 假设与实际不符"的情况

**边界：** 这个原则的代价是多一轮交互。对于简单明确的任务（比如改一个配置项），可以跳过调查直接实施。判断标准是：你对现状的确定程度是否足以支撑决策。

### 原则 5：递归一致性

**表述：** 对智能体的设计原则，应该同样适用于与智能体的协作方式。

**具体体现：**
- 我们给 ragent Agent 设计"分层信息供给"——同样用分层方式给 CC 写 TASK
- 我们给 Agent 提供"工具描述要清晰"——同样确保 TASK 中的"工具"（文件路径、数据源、参考代码）描述清晰
- 我们给 Agent 提供"行为策略"——同样在 COLLABORATION_PROTOCOL 中给 CC 提供执行原则

**这不是巧合，而是同一个问题的不同层面：** 如何让有推理能力但缺乏上下文的实体高效完成任务。

## 三、当前验证状态

| 原则 | 验证场景 | 验证程度 | 待验证 |
|------|---------|---------|--------|
| 分层信息供给 | Agent 知识库路由 | 4 个库有效 | 50+ 库是否仍适用 |
| 策略 > 信息 | system prompt 路由规则 | 行为改变显著 | 不同模型能力下是否成立 |
| 单一数据源 | 知识库白名单一致性 | 修复后稳定 | 更多组件共享数据时 |
| 先确认再行动 | ReAct 调查→实施循环 | 多轮验证有效 | 简单任务是否可以跳过 |
| 递归一致性 | TASK 设计 ↔ Agent 设计 | 模式对称 | 更复杂的多 Agent 场景 |

## 四、开放问题

这些是我们目前没有答案、但值得在后续实践中关注的问题：

1. **首次选择准确率能否提升？** 当前 Agent 仍依赖"试错+换库"的策略。是否有办法让它首次就选对？Pipeline 的意图分类器做到了（score 0.92），Agent 能否借鉴而不失去自主性？

2. **信息分层的最优切分点在哪？** 索引层放多少信息算"刚好"？我们现在的经验是每库一行 30-50 字，但这可能只是当前 4 库规模下的局部最优。

3. **策略规则会过时吗？** 当前的"低分换库"策略在 4 库场景下有效。如果库很多，盲目遍历效率很低。策略本身可能需要随规模演进。

4. **人机协作的 TASK 粒度怎么判断？** Task 009（调查+修复一体）和 Task 010（纯调查）+ Task 011（纯实施）是两种不同的粒度选择。什么时候合适合并，什么时候合适拆分？当前的判断标准是"对现状的确定程度"，但还需要更多 case 来验证。
