## 背景

### 为什么现在做这个

我们的实验有一个隐含条件——**对话都很短**（最长 Q7 也就 3 轮），上下文窗口完全够用。但真实场景中对话可能持续几十轮，工具输出可能非常长。问题从"怎么给模型好的信息"变成"**怎么在信息过载时保持信息质量**"。

CC（Claude Code）作为一个成熟的 Agent Loop 产品，必然有完善的记忆和上下文管理机制。我们不是要照搬它的实现，而是**带着自己的猜测去验证**，理解它的设计逻辑后，判断哪些对 ragent 有迁移价值。

### 我们已经有的认知基础

从五次实验中提炼的核心公式：**模型决策质量 = f(当前上下文的信息质量)**

从实验中提炼的范式：**确定性的归代码，不确定性的归模型。**

从 Raschka 文章中了解到的六组件中，我们已验证了 3 个（实时上下文、工具接入、提示词形态），还有 3 个未触及：**上下文瘦身（组件4）、会话记忆（组件5）**、子智能体委派（组件6，暂不关注）。

### 三个待验证的猜测

**猜测 A：记忆管理遵循同样的 Loop 范式——模型是决策者，代码提供工具。**

和检索一样，记忆的卸载、加载、临时存储、多级存储都可以抽象为工具。Loop 骨架不变，只是多了几个"感官"。Raschka 文章中的 `note_tool` 支持这个猜测。

**猜测 B：不是所有记忆操作都需要模型决策，边界是"确定性 vs 不确定性"。**

```
确定性（代码自动做）：每轮结果追加到完整记录、超长输出截断、重复内容去重、近期保留远期压缩
不确定性（工具让模型决定）：什么时候记笔记、记什么、什么时候回看历史
```

**猜测 C：CC 的"五层压缩策略"本质是"信息不丢弃，只换存储位置"（原则4）+ "渐进式降级"（原则5）的组合。**

五层压缩大概率是从"完全保留"到"只留引用路径"的连续光谱，触发条件和"信息被再次需要的概率"有关。

## 关键约束

1. **聚焦记忆和上下文管理** — 不要发散到 CC 的其他子系统（权限、沙箱、子智能体等），除非它们直接和上下文管理有交互
2. **带着问题找答案** — 三个核心问题是主线，不要迷失在代码细节中。如果某段代码和三个问题都不相关，跳过它
3. **代码证据优先** — 每个结论都附上文件路径和关键代码片段，不要猜测
4. **区分"确定性 vs 不确定性"** — 这是我们的分析框架，在描述每个机制时都标注它属于哪一类
5. **深度适可而止** — 理解设计逻辑和决策边界即可，不需要逐行读实现细节。能回答三个问题就够了

## 我们最想从这次探查中学到什么

1. **CC 在"确定性 vs 不确定性"的边界画在哪里？** — 这直接指导 ragent 的记忆系统设计：哪些硬编码，哪些交给模型
2. **"信息不丢弃，只换存储位置"在实践中是怎么做的？** — 这是 ragent 当前记忆机制最大的缺陷（直接丢弃旧对话）
3. **模型对自身上下文状态的感知程度** — 模型知不知道自己"忘了"什么？这决定了 ragent 是否需要在 System Prompt 中告知模型上下文被压缩了

最终目标不是复制 CC 的实现，而是理解它的设计逻辑后，用同样的"确定性归代码、不确定性归模型"范式，为 ragent 设计一个适合自己场景的记忆系统。
