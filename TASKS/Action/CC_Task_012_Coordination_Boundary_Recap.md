## 背景与边界说明

ragent 项目目前的协同调度状态如下:

- **已基本落地(可能不完善):** 意图树识别 SYSTEM 类问题后向 Agent 转交的机制
- **基于真实观察的外推设计(尚未完整落地):** 基于问题长度、关键词、子查询拆解结果的复杂度路由机制

本任务的目的不是判断"已实现的部分够不够好",而是**把"什么是已实现的、什么是设计的"这条边界画清楚**。这条边界的清晰度,直接决定了简历措辞的准确性和面试讲述的可信度。

## 任务分解

### 第一阶段:四份 Report 的事实归纳

按下列三个主题,从四份 Report 中提取相关事实(只提取,不改写,不修辞):

**主题 A:Agent 路由问题的发现与定位**
- 问题最初是怎么暴露出来的?(测试问题、观察到的现象、与 Pipeline 的对比)
- 根因定位的关键事实是什么?(工具描述太简、白名单机制、缺少切换意识等)

**主题 B:已实施的修复动作**
- 总共做了哪些修复?按时间或编号顺序列出
- 每个修复对应改了什么文件、修了什么具体问题
- 每个修复后的可验证效果(分数对比、行为对比)

**主题 C:遗留问题与后续方向**
- Report 011 末尾"仍存在的不足"那段的具体内容
- 这些不足在四份 Report 中是否还有更早的迹象(可能在 Report 006 / 010 中已经出现过)
- 当前已有的解决思路(只记录 Report 中明确写下的,不要外推)

---

### 第二阶段:已落地部分的代码点查证

**只查证一个关键点:意图树 SYSTEM 类问题向 Agent 的转交入口**

具体来说,需要在代码中确认以下事实:

1. **入口在哪里?** Pipeline 的意图分类阶段判断为 SYSTEM 类后,在哪个类、哪个方法里发起对 Agent 的调用?
2. **转交时传递了什么?** 用户原始问题?改写后的问题?上下文?
3. **Agent 接到 SYSTEM 类问题后,实际调用的工具是哪些?** (Report 011 已经提到 system_info_query 和 knowledge_search_with_rerank,需要确认是否还有其他)
4. **"基本实现但不完善"具体不完善在哪里?** (比如:有些 SYSTEM 类问题没被覆盖?转交后 Agent 行为不稳定?错误处理缺失?这部分是 CC 在阅读代码时的客观观察,不需要给出修复方案)

**查证范围限制:**
- 只看与 SYSTEM 转交直接相关的类(意图分类器、转交入口、Agent 入口)
- 不去看 Pipeline 的检索 / Rerank / 生成阶段(与本任务无关)
- 不去看 Agent 内部的工具实现细节(Report 011 已经覆盖)

---

这张表的作用:让任何人(包括你自己,也包括未来的面试官)一眼能分清哪些是已发生的事,哪些是合理但未落地的设计。

## 注意事项

- **聚焦边界,不做评价**:本任务不需要评价"修复够不够好""设计合不合理",只需要把"是什么"和"还不是什么"分清楚

- **不需要写"接下来该做什么"的建议**:那是规划者(我和操作者)的工作
