# CC Task 004：Agent 模式前端集成

> **目标：** 让用户在 ragent web 端能选择"Agent 模式"进行问答，复用现有的聊天界面和 SSE 机制。
> **原则：** 最小化改动。能复用就复用，不做额外的 UI 设计。跑通 > 完美。
> **产出：** `CC_Report_004_Frontend_Integration.md`

---

## 背景

Agent Loop 后端已跑通（Task 001-003），端点 `/api/ragent/agent/chat` 可用，返回 SSE 流式响应（`event:content` + `event:done`）。但目前只能通过 curl 访问，用户无法在前端使用。

本任务的目标是在前端加一个最简的模式切换机制，让用户能在 Pipeline 和 Agent 之间选择。

**已知问题：** Report 003 中发现路径有双重 agent（`/agent/agent/chat`），如果尚未修复，在本任务中一并处理。

## 第二阶段：最小化实施

基于调查结果，实现最简的模式切换。以下是**预期方案**，实际实现以调查结果为准：

### 方案 A（首选）：在发送区域加一个切换开关

在聊天输入框附近加一个简单的 toggle / 下拉 / 按钮，让用户选择"普通模式"或"Agent 模式"。选择 Agent 模式后，发送请求时改为调用 `/api/ragent/agent/chat` 端点。其他逻辑（SSE 接收、消息渲染）尽量复用。

### 方案 B（备选）：URL 参数区分

如果改前端 UI 成本太高，考虑在现有请求上加一个 `mode=agent` 参数，后端路由到 AgentLoop。这需要改后端的 Chat Controller 加一个分支，但前端只需加一个隐藏开关或 URL 参数。

### 方案 C（最简备选）：独立页面

如果以上都太复杂，新建一个极简的 HTML 页面，只用来测试 Agent 模式。不修改现有前端代码。

### SSE 格式兼容

如果 Agent 端点和 Pipeline 端点的 SSE 事件格式不一致，优先调整**后端 Agent 端点的输出格式**来匹配前端的期望（改后端一处比改前端多处更安全）。

### 中间过程展示

MVP 阶段**不展示 Agent 的中间过程**（工具调用、检索结果）。用户只看到最终回答，体验与 Pipeline 模式一致。后续如果需要展示"思考过程"，可以扩展 SSE 事件类型（如 `event:thinking`、`event:tool_call`），但这不在本次范围内。
