## 背景

### 当前状况

ragent 有两种问答模式：
- **Pipeline 模式**（`/api/rag/v3/chat`）：已有完整的对话持久化，用户的历史对话可以查看和恢复
- **Agent 模式**（`/api/ragent/agent/chat`）：对话可能没有保存逻辑，或保存不完整，用户离开后对话丢失

Agent 模式从 Task 002 合入以来，经历了 Task 006（Strategy Router）、Task 008（系统自省）、Task 009（知识库动态化）、Task 011（路由增强）等多轮增强，功能已经比较完整。但对话持久化这个基础设施一直没有专门处理。

### 为什么这个问题重要

对话持久化不只是"保存聊天记录"——它是后续更高级能力的基础：
- **多轮对话**需要历史上下文
- **上下文记忆/压缩**需要知道之前聊了什么
- **用户体验**的基本要求是不丢对话

但我们当前只需要解决最基本的问题：**对话能保存、能恢复、能继续**。上下文记忆和压缩是后续的事。

### 核心矛盾

Agent 模式的对话结构与 Pipeline 不同：

| 维度 | Pipeline | Agent |
|------|----------|-------|
| 交互模式 | 一问一答 | 多轮工具调用后回答 |
| 中间过程 | 无（或很少） | 工具调用历史、检索结果、rerank 分数 |
| 一次对话的内容 | 用户问题 + 系统回答 | 用户问题 + N 次工具调用 + 系统回答 |
| SSE 事件 | content + done | content + done（+ 未来可能的 tool_call） |

简单复用 Pipeline 的保存逻辑，可能只保存了"问题+最终回答"，丢失了 Agent 的执行过程。但这在当前阶段可能就够用了——执行过程的保存可以后续再加。

**判断标准：** 最小可行方案 = 用户能看到历史问答（问题+最终回答）。如果 Pipeline 的机制能做到这一点，就复用它。Agent 特有的执行报告（工具调用历史等）的持久化，标记为后续方向即可。

## 第一阶段：调查

### 1. Pipeline 的对话持久化机制

这是核心调查项，因为它决定了我们能复用多少。请搞清楚：

- **存储层：** 对话记录存在哪里？（MySQL 表？Redis？文件？）哪些表、什么字段？
- **写入时机：** Pipeline 在什么时候保存对话？（请求进来时？回答生成后？SSE 结束时？）
- **写入内容：** 保存了什么？（用户消息、系统回答、conversationId、时间戳、还有什么？）
- **读取/恢复：** 前端是怎么加载历史对话的？调哪个接口？
- **对话 vs 消息的数据模型：** 是一个 conversation 包含多个 message 的结构吗？

### 2. Agent 模式的当前保存状态

- Agent 端点（`AgentChatController` / `AgentLoopService`）中有没有任何保存对话的逻辑？
- 如果有，保存了什么？如果完全没有，确认一下
- Agent 模式的 conversationId 是怎么管理的？（前端传入？后端生成？每次新建？）

### 3. Agent 执行过程的数据

了解一下当前 Agent 的执行报告（工具调用历史等）是怎么处理的：
- 执行报告是在内存中生成后直接 SSE 推送，还是有持久化？
- AgentLoop 的中间状态（每轮的 tool_call 和 tool_result）存在哪里？

这一项不是为了现在就保存执行过程，而是为了了解全貌，在 REPORT 中标记为"后续可扩展的点"。

### 4. 实验中的相关探索

检查 `experiment/` 目录和 `FEEDBACK/` 目录中是否有关于以下主题的内容：
- 多轮对话上下文管理
- Microcompact / 上下文压缩
- 对话记忆机制

如果有，简要摘要相关发现，标注哪些设计思路可能与对话持久化相关。这些不一定现在用，但能帮助我们在设计保存结构时预留扩展空间。

## 第二阶段：方案评估与实施

基于调查结果，选择最合适的方案。以下是三个可能的方向，不限于此——如果你发现更好的方式，请采用：

### 方向 A：直接复用 Pipeline 的保存逻辑

如果 Pipeline 的持久化是通用的（比如通过拦截器、AOP、或公共 Service），Agent 端点接入即可。

**适用条件：** Pipeline 的保存机制与端点解耦，Agent 只需调用同一个 Service
**预期改动：** 在 AgentLoopService 或 AgentChatController 中，在回答生成后调用保存

### 方向 B：提取 Pipeline 的保存逻辑为公共组件，Agent 调用

如果 Pipeline 的保存逻辑耦合在 Pipeline 代码中，需要先提取为独立组件。

**适用条件：** Pipeline 的保存代码嵌在 Pipeline 流程里，不能直接调用
**预期改动：** 提取 → 公共 Service → Agent 调用

### 方向 C：Agent 独立实现（最后备选）

如果 Pipeline 的数据模型不兼容 Agent（比如消息结构完全不同），可能需要 Agent 自己的保存逻辑。

**适用条件：** Pipeline 和 Agent 的对话数据模型差异太大，复用成本高于新建
**预期改动：** 新建 Agent 专用的保存逻辑

**选择原则：** A > B > C。能复用就复用，需要适配就适配，只有不兼容才新建。但不管选哪个方案，保存的数据模型应该**预留 Agent 特有字段的扩展空间**（即使当前不填充）。比如，如果 message 表有 `metadata` 或 `extra` 字段，Agent 未来可以用它存工具调用历史。

## 验证

修复完成后的基本验证：

1. **保存验证：** Agent 模式下提一个问题，检查数据库/存储中是否有记录
2. **恢复验证：** 刷新页面或重新进入，历史对话是否能加载
3. **继续对话：** 在恢复的对话基础上继续提问，是否正常
4. **Pipeline 回归：** Pipeline 模式不受影响
