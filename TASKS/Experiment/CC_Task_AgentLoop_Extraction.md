# CC 任务：从 claude-code-sourcemap 提取 Agent Loop 最小骨架

## 背景（你需要知道的上下文）

我们正在做一个实验：**在 ragent（Java/Spring Boot RAG 系统）上实现一个 mini Agent Loop**，从"固定 pipeline"演进到"模型驱动的循环"。

当前 ragent 的在线链路是硬编码的 pipeline：
```
查询重写 → 意图分类 → 向量检索 → Rerank → 自适应截断 → LLM生成
```
每一步的顺序由代码决定，模型只在管道内部的特定节点被调用，但从不决定"下一步做什么"。

CC（Claude Code）的架构完全不同：一个 `while(true)` 循环，模型是唯一决策者——模型返回 `tool_use` 就继续循环，不返回就终止。代码不做业务决策，只负责执行和反馈。

我们已经从 DeepWiki 获得了高层理解（见 Claude_Code_Cognitive_Map.md），现在需要**深入源码提取可迁移的实现细节**。

## 核心认知：大部分代码在"驯服概率性"

我们已经识别到一个关键洞察——CC 的 agent loop 骨架本身极其简单（while + needsFollowUp），工程量巨大的部分是在驯服模型决策的概率性：权限系统、上下文压缩、预算控制、Stop Hooks、错误反馈。这些"护栏"是 pipeline 架构不需要的，因为 pipeline 的行为是确定性的。

## 你的任务：提取三个精确信息点

请在 `claude-code-sourcemap/restored-src/` 目录下定位以下内容。对每个点，我需要：
- **最小代码片段**（不超过 30 行，提取核心逻辑，去掉日志/注释/边缘情况）
- **数据流描述**（输入什么 → 经过什么判断 → 输出什么）
- **一句话祛魅**（这段代码本质上在做什么）

### 信息点 1：循环驱动的最小骨架

**目标文件：** `src/query.ts`（`queryLoop` 函数）

**提取内容：**
- `needsFollowUp` 变量从初始化到被设置为 true 的完整判断链路
- 循环继续（`continue`）和终止（`return`）的分支条件
- `state` 对象在迭代间怎么传递（哪些字段被更新，哪些不变）

**我们特别关心：** 剥离掉压缩、Hook、错误恢复等所有护栏逻辑后，纯粹的"调模型→检查是否有tool_use→执行工具→拼结果→下一轮"的骨架代码长什么样？能不能提炼成 20 行伪代码？

### 信息点 2：工具定义的序列化格式

**可能的目标文件：** 工具定义相关的 ts 文件（Tool 接口、assembleToolPool、或具体工具如 BashTool/ReadTool 的定义）

**提取内容：**
- 一个具体工具（优先选最简单的，比如 Read 或 WebSearch）的完整定义结构
- `name`、`description`、`input_schema` 是怎么组装的
- 这些定义最终怎么被序列化到 Claude API 的 `tools` 参数里

**我们特别关心：** 工具定义的数据结构是什么样的？我们在 ragent 里要把"向量检索""查询重写""最终生成"包装成工具，需要知道一个工具定义的最小必要字段。

### 信息点 3：最轻量的护栏实现

**可能的目标文件：** maxTurns 相关逻辑（query.ts 中）、或最简单的 Hook 实现

**提取内容：**
- `maxTurns` 的检查逻辑（在循环的哪个位置检查、超出后怎么处理）
- 如果 Hook 系统太复杂，就跳过，只看 maxTurns
- 工具执行失败时，错误是怎么格式化为 `tool_result` 反馈给模型的（找到 `is_error: true` 的构造位置）

**我们特别关心：** 在 ragent 的 mini agent loop 里，最低成本的护栏是什么？我们倾向于先实现 maxTurns（防无限循环）+ 错误反馈（让模型知道工具失败了），这两个加起来需要多少代码？
