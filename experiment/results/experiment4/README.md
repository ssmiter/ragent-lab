## 一、为什么要做这个实验

### 1.1 我们走到哪了

```
实验0-3:  Agent Loop 从骨架到跨库综合     → ✅ 价值跃迁验证完成
CC 源码:  记忆/压缩两轮探查              → ✅ 子系统观确立
本次:     Phase 1A — 给 Agent Loop 装上记忆基础设施
```

实验0-3 有一个隐含条件：**对话都很短**（最长 Q7 也就 3 轮）。上下文窗口完全够用，不需要任何记忆或压缩机制。

但真实场景中，用户可能围绕一个主题连续追问 15-20 轮。每轮检索返回 ~10,000-100,000 字符的结果，10 轮后光历史检索结果就有 **百万字符级别**——大部分是和当前问题无关的旧结果，但它们占着上下文窗口、消耗着模型注意力。

### 1.2 核心问题

这次实验要回答三个问题：

1. **上下文膨胀有多快？** — 用 transcript 数据量化
2. **清理旧结果会影响模型吗？** — 对比 compact 前后的回答质量
3. **应该保留什么、丢弃什么？** — 针对检索场景设计 compact 策略

## 二、本质洞察：上下文膨胀的真面目

### 2.1 数据揭示的真相

Run 1 (baseline) 的膨胀曲线：

| Turn | messages 总字符 | tool_result 字符 | tool_result 占比 |
|------|----------------|-----------------|-----------------|
| 1    | 90,164         | 89,097          | **98.8%**       |
| 2    | 422,149        | 420,582         | **99.6%**       |
| 7    | 933,518        | 926,087         | **99.2%**       |
| 15   | **1,233,375**  | **1,219,451**   | **98.9%**       |

**数据揭示的本质：**

```
tool_result 占比稳定在 98-99%
        ↓
检索结果是唯一膨胀源
        ↓
不是"对话变长导致膨胀"，而是"每次检索都在往上下文里堆料"
        ↓
解决方案：不堆那么多料就行了
```

### 2.2 为什么不是 CC 的策略

CC 是编程智能体，膨胀来自 **Read 工具返回的文件内容**。CC 的 compact 策略是"文件读过了就压缩"，因为代码文件有结构（函数名、类名），压缩后模型还能理解。

ragent 是知识问答系统，膨胀来自 **检索结果**。检索结果是碎片化的文档片段，没有代码那样的结构。但有一个关键特点：**它们是"历史"**——模型在 Turn 10 不需要 Turn 1 搜到的全文，只需要知道"Turn 1 搜过这个主题"。

所以 Microcompact 的策略是：

```
CC 的策略: 保留文件名和关键代码结构
ragent 的策略: 保留"搜过什么、从哪个库、得分多少"
```

本质相同：**保留索引信息，丢弃全文内容**。只是"索引信息"在不同场景下有不同的含义。

## 三、祛魅：Microcompact 其实很简单

### 3.1 Microcompact 的本质

Microcompact 听起来高大上，本质上就是：

```python
# 伪代码
for message in messages:
    if message.role == "tool" and message.age > threshold:
        message.content = "之前搜过这个，详情在 transcript 里"
```

**三行代码做的事情：**
1. 找到旧的 tool_result
2. 把它的内容替换成一行摘要
3. 原始内容不丢弃，只是换个地方存（transcript 文件）

### 3.2 实际效果对比

**原始内容（Turn 2 的检索结果，105,405 字符）：**

```
在知识库 [ragentdocs] 中找到 5 个相关片段：

[1] (Rerank分数: 0.86) (来源: 未知来源)
# 《AI大模型Ragent项目》——MCP之官方Java-SDK深度解析

[来自： 拿个offer-开源&项目实战](https://wx.zsxq.com/group/51121244585524)

...（后面还有 100,000+ 字符的文档全文）
```

**压缩后（68 字符）：**

```
[已压缩的搜索结果 - Turn 2]
知识库: ragentdocs, 最高分 0.86
完整记录已保存至 transcript 文件
```

**节省：105,337 字符（99.9%）**

### 3.3 为什么模型不需要旧检索的全文

这是一个关键洞察：**模型在后续轮次中，实际上不回顾旧检索结果的全文**。

观察 Run 1 的 Turn 14-15（总结题），模型直接回答，**不再调用工具**。这说明：

```
模型能综合之前讨论内容进行回答
        ↓
模型依赖的是"对话中已经讨论过的结论"，而不是"原始检索结果"
        ↓
原始检索结果在首次回答后，其价值已被提取到对话中
        ↓
压缩旧检索结果不影响模型，因为模型本来就不回头看全文
```

### 3.4 Run 2 验证了这个洞察

| Turn | Run 1 总字符 | Run 2 总字符 | Run 1 占比 | Run 2 占比 |
|------|-------------|-------------|-----------|-----------|
| 15   | **1,233,375** | **72,842** | 98.9%     | 83.2%     |

Run 2 开启 Microcompact 后：
- 总字符从 1.23M 降至 72K（**节省 94%**）
- 模型行为**没有异常**，正常回答总结题
- 模型能引用之前讨论的内容，没有表现出困惑

这验证了假设：**模型依赖的是对话中的结论，不是原始检索全文**。

## 四、已实现组件

### 4.1 TranscriptWriter.java

职责：将 Agent Loop 的完整对话记录写入 JSONL 文件。

**核心原则：**
- 完整记录，不做任何截断 — transcript 是"冷存储"
- 每轮写入后立即 flush，防止异常中断丢失
- 独立组件，即使出错也不影响主循环

**输出路径：** `experiment/results/experiment4/transcripts/{experimentId}_{timestamp}.jsonl`

### 4.2 MicrocompactProcessor.java

职责：清理旧的 tool_result 内容，保留结构但丢弃细节。

**核心逻辑：**
- 当 tool_result 的"年龄"超过阈值时（默认 3 轮），将完整内容替换为摘要
- 保留"搜了什么、从哪个库搜的、得分多少"
- 丢弃 chunk 全文（主要膨胀源）

**Compact 摘要格式：**
```
[已压缩的搜索结果 - Turn N]
知识库: ragentdocs
返回: 5 个片段, 最高分 0.95, 来源: ragent_architecture.md, ragent_overview.md 等
完整记录已保存至 transcript 文件
```

### 4.3 AgentLoopExperiment4.java

支持多轮对话的实验入口，集成 Transcript 和 Microcompact。

**运行方式：**
```bash
# Run 1 - baseline（无 compact）
mvnw exec:java -pl experiment -Dexec.mainClass=...AgentLoopExperiment4 -Dexec.args="apiKey baseline"

# Run 2 - compact（开启 compact）
mvnw exec:java -pl experiment -Dexec.mainClass=...AgentLoopExperiment4 -Dexec.args="apiKey compact"
```

## 六、结论与本质总结

### 6.1 核心结论

| 问题 | 结论 |
|-----|------|
| 上下文膨胀有多快？ | 15轮达到 1.2M 字符，tool_result 占比 98-99% |
| 清理旧结果会影响模型吗？ | **不会**，模型依赖对话中的结论，不回顾原始检索全文 |
| 应该保留什么、丢弃什么？ | 保留"搜过什么、从哪个库、得分多少"，丢弃 chunk 全文 |

### 6.2 本质总结

**Microcompact 的本质：**

```
信息不丢弃，只换存储位置
        ↓
原始内容 → transcript 文件（冷存储）
摘要信息 → 上下文窗口（热存储）
        ↓
模型不需要全文，只需要"索引"
        ↓
这不是 LLM 摘要，是纯代码逻辑
        ↓
本质是"把历史档案归档，桌上只留目录"
```

**上下文膨胀的本质：**

```
不是对话变长导致膨胀
        ↓
是每次检索都往上下文堆料
        ↓
解决方案：堆料时只堆"索引"，不堆"全文"
```

### 6.3 后续优化方向

1. **动态阈值**：根据 messages 总字符动态调整 compact 阈值
2. **Autocompact**：让 LLM 生成更智能的摘要（任务 B）✅ 已完成
3. **MEMORY.md**：跨会话的持久化记忆（任务 B）✅ 已完成

## 七、Run 3 (Autocompact) — LLM 生成摘要

### 7.1 设计思路

Microcompact 的问题：68 字符索引信息量太低——模型只知道"搜过"，不知道"搜到了什么"。

**Autocompact 的核心区别：**
- **Microcompact**：纯代码逻辑，生成 68 字符索引（"搜过什么库、得分多少"）
- **Autocompact**：调 LLM，生成 200-300 字符摘要（"搜到了什么关键发现"）

**摘要 Prompt 设计原则：**
1. 提取"这次检索最重要的发现是什么"
2. 说明"信息来自哪个知识库、哪些文档"
3. 说明这些发现与用户问题的关系（如何帮助回答问题）
4. 不要缩写原文，而是提取理解后的要点

**模型选择：**
- 主模型：qwen-max（用于 Agent Loop 多轮对话，与 Run 1/2 一致）
- 摘要模型：qwen-turbo（轻量模型，用于生成摘要）

## 八、Bug 修复记录

### 8.1 重复压缩 Bug（已修复）

**问题描述：** 初始实现中，同一 tool_result 在每个 Agent Turn 都会被重新压缩，导致 LLM 浪费调用。

**日志证据（修复前）：**
```
11:02:06.773 Autocompact tool_result (Turn 1, Age 4): 78298 chars -> 394 chars
11:02:34.972 Autocompact tool_result (Turn 1, Age 4): 394 chars -> 336 chars  # 重复压缩
```

**修复方案：**
1. 在已压缩内容中添加标记 `[Autocompact-T{turn}]`
2. 检测到标记则跳过，不再重复调用 LLM
3. SessionMemoryWriter 添加去重机制

**修复效果：**
| 指标 | 修复前 | 修复后 |
|------|-------|-------|
| LLM 调用次数 | 64 次 | **6 次** |
| 跳过已压缩 | 0 | 57 条 |
| LLM 总耗时 | 100,893 ms | **7,054 ms** |

### 8.2 摘要上下文改进（已实现）

**问题：** 摘要缺少原始问题上下文，可能与讨论主题不完全匹配。

**改进：** 在摘要 Prompt 中传入原始用户问题：
```
用户原始问题：Ragent 系统的整体架构是什么？

要求：
1. 提取"这次检索最重要的发现是什么"
2. 说明这些发现与用户问题的关系
...
```

### 8.3 成本效益分析（修复后）

| 方案 | 压缩效果 | LLM 开销 | 适用场景 |
|------|---------|---------|---------|
| Microcompact | 94% 节省 | 无 | 成本敏感、信息量要求低 |
| Autocompact | 96.9% 节省 | 6次轻量模型调用（7秒） | 质量敏感、需要信息密度 |
| 混合策略 | TBD | 折中 | 平衡成本与质量 |

## 九、结论

### 9.1 核心结论

| 问题 | 结论 |
|-----|------|
| Autocompact 效率如何？ | **极高**：修复后仅需 6 次 LLM 调用（vs 修复前 64 次），耗时 7 秒 |
| 压缩效果如何？ | **优异**：最终上下文 38,766 字符（vs baseline 1,233,375），节省 96.9% |
| 模型行为变化？ | 工具调用从 28 次降到 6 次，摘要质量足够支撑后续问答 |
| SessionMemory 有价值吗？ | **有价值**，为未来会话提供了"讨论了什么"的快速索引 |

### 9.2 本质总结

**Autocompact 的本质：**
```
用 LLM 理解能力换信息密度
        ↓
原始检索结果 → LLM 提取要点 → 200-300 字符摘要
        ↓
模型能知道"发现了什么"，而不只是"搜过"
        ↓
代价：极低的 LLM 调用（修复后仅 6 次）
```

**重复压缩修复的关键：**
```
标记 + 去重 + 状态追踪
        ↓
[Autocompact-T{turn}] 标记已压缩内容
        ↓
processedToolCallIds 集合避免重复处理
        ↓
LLM 调用从 64 次降到 6 次
```

**SessionMemory 的价值：**
```
为"未来"准备的知识沉淀
        ↓
当前会话：模型通过上下文摘要获取信息
未来会话：新模型可通过 SessionMemory 快速了解"之前讨论了什么"
        ↓
这是迈向"长期记忆"的第一步
```
