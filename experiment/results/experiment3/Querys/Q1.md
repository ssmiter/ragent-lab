## 一、项目整体结构

Ragent 采用 Maven 多模块架构，由 **四个子模块** 组成：

```
bootstrap ──依赖──▶ framework
    │                   ▲
    └──依赖──▶ infra-ai ┘

mcp-server（独立运行）
```

**依赖关系说明**：
- `bootstrap` 是最上层的业务模块，依赖 `framework`（通用基础设施）和 `infra-ai`（AI 能力层）
- `mcp-server` 是一个独立的 Spring Boot 应用，通过 HTTP 提供 MCP 工具调用服务

---

## 二、四大子模块详解

### 1. framework —— 通用基础设施层

不包含业务逻辑，放的是跨模块复用的基础组件：

| 功能模块 | 核心类 | 说明 |
|:--|:--|:--|
| 统一数据结构 | `Result<T>`、`ChatRequest`、`ChatMessage`、`RetrievedChunk` | 标准API响应、大模型请求封装、对话消息、向量检索结果 |
| 异常体系 | `BaseErrorCode`、`AbstractException`、`GlobalExceptionHandler` | 错误码枚举、异常基类、全局异常拦截 |
| 用户上下文 | `UserContext`（基于 TTL） | 跨线程池传递用户信息 |
| 链路追踪 | `RagTraceContext` | 分布式链路追踪，维护 traceId、taskId |
| SSE 工具 | `SseEmitterSender` | 线程安全的 SSE 连接管理 |

### 2. infra-ai —— AI 模型基础设施层

作为系统和大模型之间的桥梁，屏蔽不同 AI 服务商的接口差异：

| 子包 | 功能 | 核心类 |
|:--|:--|:--|
| chat | 对话能力 | `ChatClient`（接口）、`RoutingLLMService`、`StreamCallback` |
| embedding | 向量化能力 | `EmbeddingClient`、`RoutingEmbeddingService` |
| rerank | 重排序能力 | `RerankClient`、`RoutingRerankService` |
| model | 模型路由与容错 | `ModelSelector`、`ModelRoutingExecutor`、`ModelHealthStore`（熔断器） |

**核心设计**：多模型候选 + 自动降级 + 熔断保护

### 3. bootstrap —— 业务主模块

系统入口，代码量最大，按功能域划分为以下包：

| 子包 | 负责内容 |
|:--|:--|
| rag | RAG 对话核心（加载记忆→查询改写→意图识别→检索→上下文组装→流式生成） |
| ingestion | 数据 ETL 流水线（Fetcher→Parser→Enricher→Chunker→Enhancer→Indexer） |
| knowledge | 知识库管理（知识库、文档、文档块三个层级 CRUD） |
| admin | 管理后台（Dashboard、指标聚合） |
| user | 用户认证（基于 Sa-Token） |
| core | 核心工具（文本切块、文档解析） |

### 4. mcp-server —— MCP 工具服务

独立部署的 Spring Boot 应用，实现 MCP 协议：

| 层级 | 核心类 |
|:--|:--|
| 协议层 | `MCPEndpoint`（HTTP入口）、`MCPDispatcher`（方法路由） |
| 工具注册 | `MCPToolRegistry`、`MCPToolExecutor`（工具执行器接口） |
| 内置工具 | `WeatherMCPExecutor`（天气查询）、`TicketMCPExecutor`（工单查询） |

```
用户提问 → 加载记忆 → 查询改写 → 意图识别 → 路由分发

    ├── 闲聊路径：LLM直接回复
    └── 澄清路径：返回引导问题
    ↓
保存回复 → SSE推送前端
```
